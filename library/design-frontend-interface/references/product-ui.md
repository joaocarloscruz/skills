# Product Interfaces

Use these rules for dashboards, admin tools, settings, editors, forms, tables, and software people operate repeatedly.

## Design around work

- Optimize for the user's recurring verb: compare, approve, schedule, diagnose, edit, or monitor.
- Preserve context across navigation and state changes. Make location, scope, selection, and unsaved work visible.
- Give frequent actions efficient paths without hiding safe defaults from new users.
- Let information priority determine density. Dense is not cluttered when grouping and alignment are strong.

## Structure information

- Build hierarchy through position, grouping, type, and whitespace before adding cards or color.
- Use cards only when their boundaries communicate a meaningful object or elevation level.
- Make tables support scanning, comparison, sorting, filtering, selection, overflow, and narrow screens according to the task.
- Keep labels and action names consistent across controls, confirmations, notifications, and documentation.

## Model interaction states

- Place validation next to the field and explain how to recover.
- Preserve entered values after recoverable errors.
- Distinguish unavailable, unauthorized, empty, loading, stale, partial, and failed states.
- Confirm destructive or expensive actions in proportion to reversibility, not habit.
- Show progress for long operations and make cancellation or safe retry available where possible.

## Adapt responsively

- Preserve the primary task rather than mechanically stacking desktop regions.
- Prioritize essential columns and actions; provide deliberate access to secondary detail.
- Keep targets large enough for touch and avoid hover-only meaning.
- Test long names, localization expansion, zoom, software keyboards, and constrained heights.

## Quality test

Ask whether an experienced user can predict where information lives, complete the primary task efficiently, understand system status, and recover from failure without losing work.
